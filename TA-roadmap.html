Docker / Docker Compose

Mảng này đã ổn cho mục tiêu deploy app. Cần học thêm phần hardening/tối ưu image nếu muốn production kỹ hơn.

Kiến thứcĐịnh nghĩa / Dùng để làm gìChi tiếtĐiểmĐộ khóTrạng tháiCòn thiếu / Học tiếp
Container Runtime BasicsImage là artifact deploy; container là instance chạy từ image.Image là package bất biến của app/dependency/runtime; container là process được cô lập chạy từ image đó. Khi debug cần nhớ container có lifecycle riêng, filesystem tạm, network namespace riêng, logs thường đọc qua runtime/K8s chứ không SSH vào sửa tay.4/5L1Đã ổnDebug network/volume/logs sâu hơn khi gặp incident.
Image BuildThiết kế Dockerfile để đóng gói app production.Dockerfile quyết định app chạy được, build reproducible, image nhẹ và ít lỗ hổng. Cần nhớ layer cache ảnh hưởng tốc độ build, dev dependency không nên đi vào production image, CMD/ENTRYPOINT là process chính, image tag phải immutable khi deploy.4/5L2Đã ổnMulti-stage build, non-root user, layer cache, healthcheck, image scan, immutable tag.
Local Compose / RegistryCompose chạy stack local; registry lưu image để server/K8s pull.Compose dùng để dựng nhiều service local như app + DB + broker bằng một network chung; registry là nơi lưu image versioned cho CI/CD và K8s pull. Bẫy cần nhớ: local volume/hot reload khác production image, secret local không được leak vào repo.4/5L1Đã ổnCompose profiles/networks, local secret handling, private registry auth rotation.

Kubernetes / K3s

Đây là mảng đã học rất nhiều qua kind và VPS k3s. Còn thiếu phần HA/multi-node nâng cao và storage production.

Kiến thứcĐịnh nghĩa / Dùng để làm gìChi tiếtĐiểmĐộ khóTrạng tháiCòn thiếu / Học tiếp
Cluster FundamentalsCluster, node, control plane, desired state và API server.Cluster là tập node chạy workload. API server nhận YAML/desired state, control plane quyết định pod nên chạy ở đâu, kubelet trên node kéo image và giữ pod sống. Cốt lõi K8s là reconciliation: trạng thái thật lệch thì controller kéo về trạng thái mong muốn.4/5L2Đã ổnHA control plane, multi-node, datastore backup, upgrade strategy.
WorkloadsPod, Deployment, ReplicaSet, StatefulSet, Job và lifecycle workload.Pod là đơn vị chạy container; Deployment phù hợp stateless service và rollout/rollback; StatefulSet phù hợp workload cần identity/storage ổn định; Job dùng cho task chạy xong như migration. Chọn sai workload sẽ làm deploy, storage hoặc retry rất đau.4/5L2Đã ổnSidecar/init container, canary/blue-green, job cleanup, stateful operator/managed DB.
Networking / IngressService, ClusterIP, NodePort, Ingress, DNS route traffic trong/ngoài cluster.Pod IP thay đổi nên Service tạo endpoint ổn định; app gọi nhau bằng DNS service name; Ingress nhận HTTP/HTTPS từ ngoài và route theo host/path. Khi debug traffic phải đi theo chain: client -> ingress -> service -> endpoint -> pod.4/5L2Đã ổnTLS, cert-manager, host-based routing, service mesh concept, NetworkPolicy.
Config / Secrets / HealthNamespace, ConfigMap, Secret, readiness/liveness/startup probe.Namespace chia phạm vi quản lý; ConfigMap/Secret inject config vào pod; readiness quyết định pod có nhận traffic không, liveness quyết định restart, startup tránh app khởi động chậm bị kill nhầm. Probe sai có thể tạo incident restart loop.3/5L2Cần nângExternal Secrets/SOPS/Sealed Secrets, RBAC namespace, config reload, health endpoint chuẩn.
Storage / Scaling / SecurityPV/PVC, StorageClass, HPA, node scheduling và network isolation.PV/PVC giữ data sau khi pod restart, nhưng stateful DB không chạy kiểu nhiều pod cùng write chung data directory. Thực tế thường là primary nhận write, replica phục vụ read/HA, hoặc dùng managed DB ngoài cluster. Scale app pod cũng phải kiểm soát DB connection, security và network policy.2/5L3Còn thiếuStorageClass, reclaim policy, snapshots, HPA với metrics thật, taints/tolerations, NetworkPolicy.

Metrics / Logs / TracesBa tín hiệu chính để debug và hiểu health hệ thống.Metrics trả lời hệ thống đang khỏe hay không bằng số liệu; logs trả lời chuyện gì đã xảy ra; traces trả lời một request đi qua service nào và nghẽn ở đâu. Event-driven / RabbitMQ / Kafka

Kiến thứcĐịnh nghĩa / Dùng để làm gìChi tiếtĐiểmĐộ khóTrạng tháiCòn thiếu / Học tiếp
RabbitMQ FundamentalsBroker, exchange, queue, binding, routing key và message flow.RabbitMQ là broker cho async work: producer gửi message vào exchange, binding/routing key đưa vào queue, consumer lấy ra xử lý. Dùng tốt cho side effect, retry, email, notification, sync CRM; không nên dùng để che giấu model sai hoặc transaction không rõ.4/5L2Đã ổnBroker cluster/HA, managed broker, exchange type trade-off, TTL/max length/priority.
Consumer SemanticsCách consumer xử lý message: ack/nack, prefetch, competing consumers, ordering.Ack nghĩa là xử lý xong mới xác nhận; nack/requeue quyết định retry; prefetch giới hạn số message đang xử lý để tránh consumer ôm quá nhiều. Scale nhiều consumer tăng throughput nhưng có thể làm mất ordering toàn cục, nên phải hiểu ordering theo queue/key.4/5L2Đã ổnPoison message, nack requeue false, throughput tuning, ordering khi scale consumer.
Reliability PatternsOutbox, retry/DLQ, idempotency và message ordering để tránh mất/trùng side effect.Outbox giải quyết DB commit thành công nhưng publish event fail; DLQ giữ message lỗi để điều tra/replay; idempotency chống xử lý trùng. Cần nhớ message system thường là at-least-once, nên consumer phải chịu được duplicate.4/5L3Đã ổnBackoff, stuck lock recovery, DLQ replay tool, outbox/DLQ alerts, Kafka partition by aggregate id.
Projection / Event SourcingConsumer build read model từ event; event sourcing lưu state changes bằng event log.Projection là consumer đọc event để build read model/report/search table; source of truth vẫn có thể là OLTP DB. Event sourcing thì event log mới là source of truth, state hiện tại được replay ra. Hai thứ này khác nhau, đừng gọi mọi event-driven là event sourcing.3/5L3Cần luyệnBuild projection table thật, replay, snapshot, event store versioning.
Kafka BasicsTopic, partition, consumer group, offset và schema registry cho event stream.Kafka phù hợp event stream throughput cao và replay; ordering giữ trong một partition, thường chọn key như orderId/userId để cùng aggregate đi cùng partition. Consumer group chia partition cho consumer; offset là vị trí đã đọc, không giống queue xóa message ngay.2/5L3Chưa practiceTạo topic, produce/consume local, consumer group/rebalance/offset commit, schema compatibility.

Clean Architecture

Kiến thứcĐịnh nghĩa / Dùng để làm gìChi tiếtĐiểmĐộ khóTrạng tháiCòn thiếu / Học tiếp
Layered BoundariesTách domain, application, infrastructure và interface để business không phụ thuộc framework/DB/HTTP.Domain chứa business rule, application điều phối use case, infrastructure nói chuyện DB/broker/API, interface nhận HTTP/GraphQL. Mục tiêu không phải tạo nhiều folder, mà giữ rule quan trọng không bị dính TypeORM/Nest/transport để dễ test và thay đổi.4/5L2Đã ổnLuyện domain modeling nhiều case, chuẩn hóa folder/naming trong team lớn, practice REST + GraphQL song song.
Use Case / Transaction BoundaryUse case điều phối flow; transaction boundary quyết định operation nào phải atomic.Use case trả lời một hành động business diễn ra qua các bước nào; transaction boundary trả lời bước nào phải cùng commit/rollback. Nếu boundary quá rộng sẽ giữ lock lâu; quá hẹp sẽ dễ có data nửa vời như order tạo rồi outbox/event không có.4/5L3Đã ổnGiữ use case vừa đủ, phân biệt orchestration/domain logic, học isolation/deadlock sâu hơn ở PostgreSQL.
Ports / Adapters / RepositoryPort là interface app cần; adapter là implementation; repository load/save aggregate và giấu ORM.Port nên xuất hiện khi application cần một capability ổn định như PaymentPort/EventPublisher/ProductCatalog; adapter là implementation cụ thể. Repository nên load/save aggregate, không biến thành nơi chứa mọi query report. Tạo port vô tội vạ sẽ làm code rối.4/5L2Đã ổnKhông tạo port vô tội vạ; command repo vs query/report repo; translate external error/model sạch hơn.
Mapping / API BoundaryMapper, presenter, DTO, error model giữ domain/ORM không leak ra API.Mapping/API boundary chuyển đổi domain, ORM model, DTO và response contract để không leak internal model ra ngoài.4/5L2Đã ổnPresenter riêng REST/GraphQL, error code contract, validation groups/nested DTO.
Policy / AuthorizationGuard chặn boundary; policy xử lý rule quyền sâu hơn theo resource/context.Authorization policy quyết định actor nào được làm action nào trên resource nào, thường sâu hơn guard/decorator đơn giản.3/5L2Cần luyệnABAC, permission theo resource/tenant, policy testing.
Advanced App PatternsUnit of Work và CQRS là pattern dùng khi complexity đủ lớn.Unit of Work gom nhiều repository trong một transaction; CQRS tách command model và query model khi read/write có nhu cầu khác nhau.2/5L3Biết trade-offHọc lại khi nhiều repo cần chung transaction hoặc khi query/report model tách khỏi command model.

Domain Model Building BlocksEntity, value object, factory và invariant biểu diễn business model sạch khỏi ORM/framework.Entity có identity và lifecycle; value object đo/diễn tả giá trị như Money/Email; factory tạo object hợp lệ; invariant là rule không được vi phạm. Domain Policy / ServiceTách rule không thuộc tự nhiên vào entity hoặc cần reuse/compose.Domain policy/service gom rule không tự nhiên thuộc về một entity hoặc cần compose/reuse qua nhiều flow.3/5L3Cần luyệnPhân biệt domain service với application service; rule composition/testing.

Schema FundamentalsTable/row/column, primary key, foreign key, nullable/default và kiểu dữ liệu nền.Schema fundamentals là cách biến dữ liệu business thành cấu trúc lưu trữ: type đúng để tránh sai dữ liệu, PK để định danh, FK để giữ quan hệ, NULL/default để biểu diễn trạng thái thiếu dữ liệu. Bẫy thường gặp là dùng kiểu quá lỏng, thiếu FK/unique, hoặc để app tự bảo vệ mọi thứ.3/5L1Đã học nềnPractice schema thật với payments/orders; học composite key, ON DELETE/UPDATE, generated column khi cần.
JSONBLưu dữ liệu linh hoạt dạng JSON binary, phù hợp metadata/event/raw payload hơn core field.JSONB hợp với metadata, raw payload, audit/event data có shape thay đổi. Data ModelingBiến business requirement thành table, relationship, rule, query support và risk.Data modeling là bước dịch business thành entity, relationship, cardinality, source of truth, snapshot khi cần, constraint và query cần support. Không phải lúc nào cũng snapshot; chỉ snapshot khi business cần giữ trạng thái tại thời điểm cũ như giá lúc mua. Schema / Access ControlSchema namespace, role/user, GRANT/REVOKE, schema/table/column permission, default privileges.Access control tách quyền theo schema/table/column/role. Runtime user của service chỉ nên có quyền cần thiết để query/write business data; migration user mới có quyền ALTER/CREATE/DROP. Cần nhớ GRANT USAGE ON SCHEMA chỉ cho đi vào schema, chưa có quyền SELECT table.3/5L2Đã practiceRole inheritance, group role, service boundary, reporting/read-only user, view-based access, default privileges với migration user.
ConstraintRule DB tự bảo vệ data: NOT NULL, UNIQUE, CHECK, FK, partial unique.Constraint là lớp bảo vệ cuối cùng cho data integrity: NOT NULL cho required, UNIQUE cho không trùng, FK cho quan hệ, CHECK cho rule đơn giản, partial unique cho case như soft delete. Rule business quá phức tạp/external dependency nên ở app, nhưng invariant dữ liệu rõ ràng nên để DB giữ.3/5L2Đang luyệnPractice constraint-first schema design trên case thật; học production migration khi thêm constraint vào bảng đã có data.
Schema ReviewNhìn schema draft và phát hiện lỗi model: thiếu bảng, sai cardinality, snapshot sai chỗ, constraint thiếu.Schema review là soi ngược từ business/query về model: có thiếu join table không, cardinality đúng chưa, source of truth nằm đâu, có duplicate nguy hiểm không, constraint nào cần DB bảo vệ, query/report có support được không. Đây là kỹ năng TA sát thực tế hơn chỉ viết SQL.2/5L2Đang luyệnLuyện dạng review ngược nhiều hơn vì sát công việc TA.
Transactions / ConcurrencyACID, transaction boundary, isolation level, pessimistic/optimistic lock.Transaction giúp nhiều thay đổi cùng thành công hoặc cùng rollback; concurrency xử lý nhiều request chạm cùng data. Cần hiểu lock row/table, lost update, race condition, deadlock. Ví dụ hot product inventory sẽ bị serialize trên cùng row dù DB có nhiều process, vì consistency bắt buộc.3/5L3Cần sâuIsolation anomalies, read committed/repeatable read/serializable, NOWAIT, SKIP LOCKED, deadlock, optimistic lock implementation.
Query / PerformanceJOIN, aggregation, index, EXPLAIN/query plan và read/report pattern.Query performance bắt đầu từ planner: PG dùng statistics từ ANALYZE, selectivity và cost để chọn seq scan/index scan/bitmap scan. Index chỉ mở thêm access path, không ép DB dùng. Đã practice covering index: ON (...) là cột dùng để filter/order, INCLUDE (...) là cột lưu kèm để trả dữ liệu từ index; nếu visibility map tốt thì có thể ra Index Only Scan với Heap Fetches thấp/0. Trade-off: covering index lớn hơn, tốn disk và làm write nặng hơn, nên chỉ dùng cho query hot path.3/5L3Đang họcĐể sau: SQL performance debugging checklist hardcore. Trước mắt chỉ cần biết index/filter/order/covering; sau này học composite index order, GROUP BY/window, join plan, sort spill và đọc EXPLAIN (ANALYZE, BUFFERS) trên query thật.
Schema MigrationVersioned schema change chạy cùng code release.Schema migration là thay đổi DB theo version thay vì sửa tay production. Production cần backward-compatible rollout: add nullable column, backfill bằng job/batch, app dual-write/switch-read, rồi cuối cùng mới enforce/drop. CREATE INDEX CONCURRENTLY build index trên bảng lớn mà vẫn cho write chạy. CHECK ... NOT VALID là cầu tạm để chặn data mới sai trong lúc data cũ chưa sạch; sau backfill thì VALIDATE CONSTRAINT, chuyển sang ALTER COLUMN ... SET NOT NULL nếu đây là required column, rồi drop CHECK nếu đã có NOT NULL thay thế.3/5L3Đang họcLuyện thêm rollback strategy, migration lock, large-table backfill, PITR/backup policy và zero-downtime checklist.
Database Bottleneck / ScalingNhận diện bottleneck DB và chọn đúng kỹ thuật giảm tải.Primary DB không phải một process duy nhất, nhưng thường là một node/instance nhận write chính; bottleneck đến từ CPU, disk I/O, WAL, lock contention, connection quá nhiều, query sai index hoặc hot row. Map xử lý: query chậm -> index/EXPLAIN; connection nhiều -> pool/PgBouncer; read nhiều -> cache/read replica; side effect nặng -> event-driven/queue; hot row -> concurrency control/data redesign; report nặng -> read model/OLAP.2/5L3Đang họcPractice đọc DB metrics, lock wait, connection pool sizing, cache invalidation, read replica lag, queue backpressure.
Large Data / ReportingPartitioning, materialized view, backup/restore phục vụ bảng lớn và reporting.Large data/reporting xử lý khi bảng quá lớn hoặc query đọc nặng làm ảnh hưởng OLTP. Partition giúp chia dữ liệu theo time/tenant/key; materialized view/read model giúp report nhanh hơn; OLAP/data warehouse tách workload phân tích khỏi DB giao dịch chính.2/5L3Còn thiếupg_dump/restore/PITR, native partitioning, materialized view/read model/report refresh.
